How to Deploy Multiple Sites from One Repo with Cloudflare Pages

2026-03-04

You have a monorepo with multiple static sites in subdirectories. You want each site deployed to its own custom domain. Cloudflare Pages can do this — you create one Pages project per site, each pointing to a different directory in the same repo.

Prerequisites

  • A GitHub repo with your sites in subdirectories (see How to Consolidate Multiple Static Sites into One Monorepo)
  • A Cloudflare account (free tier works)
  • Your domains added to Cloudflare DNS

The Setup

Say your repo looks like this:

your-repo/
├── sites/
│   ├── main/        ← yourdomain.com
│   ├── client-a/    ← client-a.com
│   ├── client-b/    ← client-b.com
│   └── client-c/    ← client-c.com

You'll create 4 separate Cloudflare Pages projects, all connected to the same GitHub repo. Each one serves a different subdirectory.

Step 1: Create the First Pages Project

  1. Go to Cloudflare Dashboard → Workers & Pages → Create
  2. Select Pages and connect your GitHub repo
  3. Set Project name (e.g. your-main-site)
  4. Under Build settings:
    • Build command: leave empty (static site, no build step)
    • Build output directory: sites/main
  5. Deploy

Step 2: Add Your Custom Domain

  1. In the Pages project, go to Custom domains
  2. Add your domain (e.g. yourdomain.com)
  3. If the domain is already on Cloudflare DNS, the CNAME record is added automatically
  4. If not, you'll need to add a CNAME record pointing to your-main-site.pages.dev

Step 3: Repeat for Each Site

Create a new Pages project for each site. The key: each project connects to the same repo but uses a different build output directory.

# Project 1
name: your-main-site
build output: sites/main
domain: yourdomain.com

# Project 2
name: client-a
build output: sites/client-a
domain: client-a.com

# Project 3
name: client-b
build output: sites/client-b
domain: client-b.com

# Project 4
name: client-c
build output: sites/client-c
domain: client-c.com

How Deploys Work

Every push to your repo triggers a build for all Pages projects connected to it. Since there's no build step (static files), each deploy takes seconds.

This means a change to sites/client-a/ also triggers a deploy for sites/client-b/. This is harmless — unchanged files deploy instantly — but it counts toward your 500 builds/month on the free tier.

Limits to Know

  • 500 builds/month on free tier. With 4 sites, every push uses 4 builds. That's ~125 pushes/month
  • 20,000 files per project. Each project only counts files in its output directory
  • 25 MB max file size for individual files
  • Unlimited bandwidth on all plans

After Migration

Once everything is live on Cloudflare Pages:

  • Disable GitHub Pages on all repos
  • Make your monorepo private — Cloudflare Pages keeps working with private repos
  • Archive the old individual repos as backup
